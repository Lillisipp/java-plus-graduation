# Explore With Me

## Описание проекта

**Explore With Me** — сервис для публикации, поиска и управления событиями.  
На данном этапе проект был преобразован из монолитного приложения в набор микросервисов. Такое разделение позволило повысить масштабируемость системы, упростить сопровождение кода, снизить связанность между частями приложения и подготовить систему к более устойчивой работе при росте нагрузки.

Все внешние запросы к системе выполняются через **API Gateway** на порту `8080`.  
Клиентам не требуется знать внутреннюю структуру приложения: маршрутизация запросов в нужный микросервис выполняется внутри системы.

---

## Архитектура проекта

Проект состоит из инфраструктурных компонентов и бизнес-микросервисов.

### Инфраструктурные сервисы

#### `gateway-server`
Единая точка входа в систему.  
Принимает все внешние HTTP-запросы и перенаправляет их в соответствующий микросервис. Благодаря этому внешний API остаётся единым, несмотря на внутреннее разделение приложения на несколько сервисов.

#### `config-server`
Сервис централизованной конфигурации.  
Все микросервисы получают настройки из Config Server, что упрощает управление конфигурацией и позволяет хранить параметры запуска в одном месте.

#### `discovery-server`
Сервис обнаружения на базе **Eureka**.  
Каждый микросервис регистрируется в Eureka и может находить другие сервисы по имени без жёсткой привязки к конкретному хосту и порту.

---

### Бизнес-микросервисы

#### `event-service`
Сервис управления мероприятиями.  
Отвечает за:
- создание и обновление событий;
- получение событий по id;
- публичный поиск событий;
- работу с категориями;
- работу с подборками событий.

#### `request-service`
Сервис управления заявками на участие.  
Отвечает за:
- создание заявки на участие;
- отмену заявки;
- подтверждение и отклонение заявок;
- получение заявок пользователя;
- получение заявок по событию;
- подсчёт количества подтверждённых заявок.

#### `user-service`
Сервис управления пользователями.  
Отвечает за:
- создание пользователя;
- удаление пользователя;
- получение списка пользователей;
- внутренние операции для проверки существования пользователя и получения пользователей по списку id.

#### `comment-service`
Сервис управления комментариями.  
Отвечает за:
- создание комментариев;
- изменение и удаление комментариев;
- получение комментариев;
- внутренние операции для получения агрегированных данных по комментариям.

#### `stats-server`
Сервис статистики.  
Используется для сохранения и получения информации о просмотрах endpoint’ов и событий.

## Взаимодействие сервисов

Внешние запросы проходят через `gateway-server`.  
Внутреннее взаимодействие между микросервисами реализовано с помощью:

- **OpenFeign** — для HTTP-вызовов между сервисами;
- **Eureka** — для поиска сервисов по имени;
- **Config Server** — для централизованного получения конфигурации.

Основные сценарии взаимодействия:
- `event-service` может запрашивать количество подтверждённых заявок у `request-service`;
- `event-service` может получать агрегированные данные о комментариях из `comment-service`;
- `event-service` может получать данные по просмотрам из `stats-server`;
- `request-service` и `comment-service` могут обращаться к `user-service` для проверки существования пользователя;
- `request-service` и `comment-service` могут обращаться к `event-service` для проверки существования события и получения краткой информации по нему.

Во внутренних взаимодействиях используется подход с **фиксированным количеством запросов**, чтобы избежать проблемы **N+1** уже на уровне межсервисного общения.  
Например, вместо отдельных вызовов по каждому `eventId` сервисы используют batch-endpoint’ы, принимающие список идентификаторов.

---

## Разделение данных между сервисами

Каждый микросервис владеет только своими данными:

- `user-service` — пользователи;
- `event-service` — события, категории, подборки, локации;
- `request-service` — заявки на участие;
- `comment-service` — комментарии;
- `stats-server` — статистика просмотров.

Связи между сервисами реализованы не через внешние ключи между базами данных разных сервисов, а через хранение идентификаторов (`userId`, `eventId`) и вызовы внутренних API.  
Такой подход уменьшает связанность сервисов и позволяет развивать их независимо друг от друга.
